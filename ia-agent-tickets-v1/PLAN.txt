# Plan: ia-agent-tickets-v1

## Objetivo
Sistema de tickets para mesa de ayuda con agente IA (LangGraph + LangChain) integrado
por rol. El agente actúa como intermediario conversacional: guía al creador para abrir
tickets, asiste al resolutor para consultar y cerrar los suyos, y acompaña al supervisor
en el monitoreo y reasignación.

---

## Estructura del Proyecto
```
ia-agent-tickets-v1/
├── backend/          (Express + SQLite — fuente de verdad de los tickets)
├── agent/            (Python — microservicio LangGraph/LangChain)
└── frontend/         (React + Vite — UI por rol + chat de agente)
```

## API Endpoints — Backend Express (Node.js :3001)

### Users
- `GET  /api/users`           — Listar todos los usuarios
- `GET  /api/users/:id`       — Ver un usuario
- `GET  /api/users?rol=X`     — Filtrar por rol

### Tickets
- `GET  /api/tickets`                — Listar tickets (con filtros: estado, urgencia, asignado_a, created_by…)
- `POST /api/tickets`                — Crear ticket (rol: creador)
- `GET  /api/tickets/:id`            — Ver ticket detallado
- `PUT  /api/tickets/:id`            — Actualizar campos base (rol: creador / supervisor)
- `PUT  /api/tickets/:id/assign`     — Asignar ticket (rol: supervisor)
- `PUT  /api/tickets/:id/resolve`    — Resolver ticket (rol: resolutor)
- `PUT  /api/tickets/:id/reopen`     — Reabrir ticket (rol: supervisor)

### History
- `GET  /api/tickets/:id/history`    — Historial completo del ticket

### Nota sobre el agente
El agente Python llama directamente al backend Express con los headers
x-user-id / x-user-rol para simular al usuario actual. No se requieren
endpoints adicionales en Express.

## Microservicio Agente IA — agent/ (Python :8000)

### Stack
- **Python 3.11+**
- **LangGraph** — orquestación del grafo ReAct (ciclo llm → tool → llm)
- **LangChain** — abstracciones de LLM y herramientas (@tool)
- **InMemorySaver** (`langgraph.checkpoint.memory`) — checkpointing por thread_id
- **FastAPI** — servidor HTTP liviano que expone el endpoint de chat
- **httpx** — cliente HTTP async para llamar al backend Express desde los Tools

### Endpoint único
```
POST /agent/chat
Body: { "user_id": 1, "user_rol": "creador", "message": "Quiero abrir un ticket" }
Response: { "reply": "...", "thread_id": "user-1" }
```

### Arquitectura del Grafo (LangGraph — patrón ReAct)

Se usa un único grafo ReAct estándar. El LLM decide qué Tools invocar
basándose en el system prompt (que varía por rol) y los mensajes del usuario.
NO se usa un nodo router custom; LangGraph ya maneja el ciclo de forma nativa.

```
 ┌──────────────────────────────────────────────┐
 │             Grafo ReAct (único)              │
 │                                              │
 │  [START]                                     │
 │     │                                        │
 │     ▼                                        │
 │  [llm_call]  ← system prompt dinámico        │
 │     │          según user_rol                │
 │     ▼                                        │
 │  [should_continue?]                          │
 │     │         │                              │
 │     │ sí      │ no                           │
 │     ▼         ▼                              │
 │  [tool_node] [END → reply al frontend]       │
 │     │                                        │
 │     └──→ [llm_call]  (ciclo)                 │
 │                                              │
 └──────────────────────────────────────────────┘

El system prompt se inyecta dinámicamente al inicio de cada invocación:
- Rol "creador"   → prompt que guía la recolección de datos para el ticket
- Rol "resolutor" → prompt que muestra tickets asignados y guía la resolución
- Rol "supervisor"→ prompt que da visión global y permite reasignar (fase 2)
```

Ventaja del patrón ReAct vs router custom:
- Menos código, menos nodos que mantener
- El LLM ya sabe qué Tool usar según el prompt y la conversación
- LangGraph maneja el ciclo llm→tool→llm de forma nativa
- Agregar Tools nuevas no requiere modificar el grafo

### Tools registradas en el agente

| Tool                  | Descripción                                                 | Llama a                            | Roles que la usan     |
|-----------------------|-------------------------------------------------------------|------------------------------------|-----------------------|
| `create_ticket`       | Crea un ticket con los datos recolectados en el chat        | POST /api/tickets                  | creador               |
| `get_created_tickets` | Devuelve los tickets creados por el usuario actual          | GET /api/tickets?created_by={id}   | creador               |
| `get_my_tickets`      | Devuelve los tickets asignados al resolutor actual          | GET /api/tickets?asignado_a={id}   | resolutor             |
| `get_ticket_detail`   | Devuelve el detalle completo de un ticket específico        | GET /api/tickets/{id}              | creador, resolutor    |
| `resolve_ticket`      | Marca un ticket como resuelto con el texto de resolución    | PUT /api/tickets/{id}/resolve      | resolutor             |
| `get_all_tickets`     | Lista todos los tickets (solo supervisor)                   | GET /api/tickets                   | supervisor            |
| `assign_ticket`       | Asigna un ticket a un resolutor (solo supervisor)           | PUT /api/tickets/{id}/assign       | supervisor            |
| `reopen_ticket`       | Reabre un ticket resuelto (solo supervisor)                 | PUT /api/tickets/{id}/reopen       | supervisor            |
| `get_resolutores`     | Lista los usuarios con rol resolutor disponibles            | GET /api/users?rol=resueltor       | supervisor            |

Nota sobre `create_ticket`: La Tool debe recolectar TODOS estos campos antes de
ejecutarse (el system prompt del creador guía esta recolección):
- tipo_requerimiento (Incidente / Solicitud / Problema)
- categoria (Hardware / Software / Red / Seguridad / Otro)
- descripcion (texto libre — obligatorio)
- urgencia (baja / media / alta / critica)
- impacto (bajo / medio / alto)  → el LLM puede inferirlo
- prioridad (baja / media / alta / urgente) → el LLM puede inferirla

### Flujo de conversación: ROL CREADOR

1. Agente saluda: "Hola {nombre}, soy tu asistente de mesa de ayuda. ¿Qué problema tienes?"
2. El usuario describe su problema en lenguaje natural
3. El agente extrae tipo_requerimiento de la descripción (o pregunta si es ambiguo)
4. El agente identifica la categoría (o pregunta si no es clara)
5. El agente pide más detalles si la descripción es vaga
6. El agente pregunta la urgencia: "¿Con qué urgencia necesitas esto?"
7. El agente infiere impacto y prioridad basándose en urgencia + descripción
8. Muestra resumen y pide confirmación: "Voy a crear el ticket con estos datos: ..."
9. Si confirma → llama `create_ticket` → "Ticket #X creado exitosamente"
10. Si el usuario quiere ver sus tickets → llama `get_created_tickets`

### Flujo de conversación: ROL RESOLUTOR

1. Agente saluda y automáticamente llama `get_my_tickets`
2. Muestra lista: "Tienes N tickets asignados: #1 Hardware (urgente), #3 Red (media)..."
3. El usuario puede preguntar por un ticket → llama `get_ticket_detail`
4. El usuario dice "quiero resolver el #3" → agente pide la resolución
5. El usuario describe la solución → agente confirma
6. Si confirma → llama `resolve_ticket` → "Ticket #3 marcado como resuelto"

### Flujo de conversación: ROL SUPERVISOR (Fase 2)

1. Agente llama `get_all_tickets` y muestra resumen por estado
2. Supervisor puede preguntar por tickets específicos o filtrar por resolutor
3. Puede reasignar via chat → llama `get_resolutores` + `assign_ticket`
4. Puede reabrir tickets resueltos → llama `reopen_ticket`

### System Prompts por rol (agent/prompts.py)

Cada rol tiene un system prompt que define:
- Identidad del agente ("Soy el asistente de mesa de ayuda de la empresa")
- Capacidades según el rol (qué Tools puede usar)
- Instrucciones de recolección de datos (para el creador)
- Formato de presentación de datos (para el resolutor)
- Restricciones (NO inventar datos, siempre confirmar antes de crear/resolver)
- Valores válidos para cada campo (extraídos del schema de la DB)

### Memoria (LangGraph InMemorySaver)

- Cada usuario tiene un `thread_id = f"user-{user_id}"`
- `InMemorySaver` (importado de `langgraph.checkpoint.memory`) persiste el
  historial del grafo en memoria RAM durante desarrollo
- En producción reemplazar por `SqliteSaver` o `PostgresSaver`
- El historial permite al agente recordar el contexto dentro de la misma sesión
  (ej: el creador empezó a describir un problema pero no terminó aún)

```python
from langgraph.checkpoint.memory import InMemorySaver

checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)

# Cada invocación usa el thread_id del usuario
graph.invoke(
    {"messages": [{"role": "user", "content": user_message}]},
    {"configurable": {"thread_id": f"user-{user_id}"}},
)
```
